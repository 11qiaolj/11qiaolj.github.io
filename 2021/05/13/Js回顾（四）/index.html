Js回顾（四）

JavaScript异步，promise，async await,事件循环，宏任务 微任务

js执行机制 event loop

js的执行机制是单线程，由上到下顺序执行的，这就造成了一个问题：前面的代码如果未完成，那么就会阻塞后面的代码。
如：出现网络请求，定时任务，DOM操作这样的需要时间来执行的任务，就会使之后的代码无法执行。

为解决这个问题，js将这些不能立即完成的任务划分为异步任务，异步任务会被放进异步任务队列中，等待所有同步任务完成。
当js遇到异步任务时，js会将这些任务放到异步队列中，然后继续执行下面的代码。
而异步任务如果执行完毕，就会将回调函数放入回调队列，等待执行（分为宏任务和微任务，之后会说）
当所有的同步任务执行完毕，就会不断去回调队列中寻找是否有回调函数需要执行。如果有，进入主线程，执行。
如果回调函数仍是异步的，就会再次进入异步栈中，重复之前的逻辑。
这就是事件循环（event loop）。

回调地狱

当我们需要用到一个异步任务所返回的结果时（如：网络请求），由于同步代码会优先执行，这样就会拿不到请求结果。
我们就需要把同步代码放在异步任务的回调函数中，等待异步任务执行完成。

这样将代码放入回调函数的方式，如果逻辑过于复杂，就会有这种不断回调的代码出现。
这样的代码难以维护，代码臃肿且可读性差，这种情况被称为回调地狱。

Promise

由于回调地狱影响了代码的可维护性，在ES6中提出了Promise的异步解决方案。
Promise采用.then() .catch()的链式操作，以resolve和reject两种回调函数处理异步操作返回的结果。

三种状态

padding：进行中
fulfilled：已成功
rejected：已失败
需要注意的是：
（1）对象的状态不受外界影响。只有异步操作的结果，可以决定当前是哪一种状态，任何其他操作都无法改变这个状态。这也是Promise这个名字的由来，它的英语意思就是“承诺”，表示其他手段无法改变。

（2）一旦状态改变，就不会再变，任何时候都可以得到这个结果。Promise对象的状态改变，只有两种可能：从pending变为fulfilled和从pending变为rejected。只要这两种情况发生，状态就凝固了，不会再变了，会一直保持这个结果，这时就称为 resolved（已定型）。如果改变已经发生了，你再对Promise对象添加回调函数，也会立即得到这个结果。这与事件（Event）完全不同，事件的特点是，如果你错过了它，再去监听，是得不到结果的。

基本用法：

Promise 新建后就会立即执行。
Promise构造函数接受一个函数作为参数，该函数的两个参数分别是resolve和reject。它们是两个函数，由 JavaScript 引擎提供，不用自己部署。

resolve函数的作用是，将Promise对象的状态从“未完成”变为“成功”（即从 pending 变为 resolved），在异步操作成功时调用，并将异步操作的结果，作为参数传递出去；reject函数的作用是，将Promise对象的状态从“未完成”变为“失败”（即从 pending 变为 rejected），在异步操作失败时调用，并将异步操作报出的错误，作为参数传递出去。

Promise实例生成以后，可以用then方法分别指定resolved状态和rejected状态的回调函数。

then方法可以接受两个回调函数作为参数。第一个回调函数是Promise对象的状态变为resolved时调用，第二个回调函数是Promise对象的状态变为rejected时调用。这两个函数都是可选的，不一定要提供。它们都接受Promise对象传出的值作为参数。

catch方法是.then(null, rejection)或.then(undefined, rejection)的别名，用于指定发生错误时的回调函数。

链式操作

Promise的最大优点就是，将难以维护的回调形式的异步操作变为链式异步操作。

Promise的静态方法

Promise存在四个静态方法
1.resolve()
let p=Promise.resolve('success')会返回一个成功状态的promise对象
且p.then(res=>{})里的res为传入的参数’success’
2.reject()
与resolve类似
let p=Promise.reject('fail')会返回一个失败状态的promise对象
且p.catch(res=>{})里的res为传入的参数’fail’
3.all()
all方法接收一个数组，数组中的值为promise对象，数组中的promise都会执行。
Promise.all([p1,p2,p3]).then(res=>{})中的res为三个promise返回的信息。
要注意的是，当几个promise中有一个出现错误，那么就会直接进入到.catch(err=>{})中。
4.race()
与all相应，不同的是，当有一个promise完成，则只会返回最先完成的promise返回的信息。

async await

上面的promise使用链式操作，可是这仍不能完全满足我们的编程需求。
链式操作的每一步需要返回一个promise对象，且仍不能像我们的同步代码一样“优雅”的出现在代码中。
于是出现了异步操作终极解决方案 async await

async关键字写在function之前，会返回一个Promise对象，并将返回值作为Promise对象的信息。
在async函数中，使用await关键字，await后面的异步操作会按照执行顺序执行，只有await后面的代码执行完，才会继续向下执行。
这样的操作方式将异步操作等同于同步代码放在函数体中，异步操作再也不需要担心可读性问题了。

async与promise并不互斥，而是要结合使用。
await相当于promise中的then，当发生错误时，await不会执行。
想要避免这种情况，要用try catch来捕获错误。

一些注意点

.then()并不一定会返回一个resolved状态的Promise对象，当then中报错，返回一个rejected状态的Promise
.catch()同样会返回一个resolved状态的Promise

async await本质上是语法糖。
js还是单线程的，还是异步，还是要event loop。

await语句的后面的代码，相当于callback，不回立即执行，会等待同步代码执行完成后执行。

宏任务&微任务

再看js执行机制

之前描述的event loop机制其实并不完整。
当主线程任务执行完毕，会先尝试触发DOM渲染，然后再触发event loop机制。
触发eventloop机制会将一个回调放入主线程中，主线程将其执行。
这样又会触发上面的条件——>主线程空——>尝试DOM渲染——>触发eventloop。
然而在这之中，还缺少了一个步骤：就是所有微任务会在DOM渲染前触发，在DOM渲染后，会执行一个宏任务，然后再次开启循环。

由此我们得知，微任务会在宏任务执行之前完成。

为什么？

微任务是由ES组织制定的，而宏任务是由W3C制定的。
微任务会进入micro task queue中而不是callback队列中


宏任务macroTask

微任务microTask

补充

在面试中面试官问起一些问题，又去查了一下。

promise.finally()后跟.then()还能不能执行?

先说finally，不论promise的状态是resolved还是rejected。finally()的回调函数都会执行。

然而，finally后的.then或.catch还能不能执行呢？
finally会根据回调的执行成否，返回一个resolved或rejected状态的promise。
因此，后面的.then仍然可以执行。

为什么要分then和catch

上面我们知道，catch其实是语法糖，本质相当于then(null,err=>{})。
因此，这个问题换句话说就是then的第二个参数和catch的区别。
那么为什么要有catch呢？直接在then中加两个参数不就好了吗
答案是，catch不仅可以捕获promise中的错误，还能捕获到then中回调函数发生的错误！

Promise.allSettled

Es2020提出一个新的promise方法 Promise.allSettled()
与Promise.all()类似，不同的是，Promise.allSettled()不论里面的promise的状态是成功的还是失败的，都会变成一个resolved的promise而执行.then()里面的操作。
